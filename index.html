About Me

Why I Build Robots

From Curiosity to Engineering

I am a robotics engineer specializing in full-stack architecture, focusing on the seamless integration of mechanical hardware, control software, and artificial intelligence. I am driven by the moment these disciplines converge — whether it's developing autonomous navigation systems or designing custom neural networks for adaptive flight simulations. I don't just write code; I build complete systems that turn complex technical challenges into functional, real-world solutions.

I am currently pursuing my MSc in Robotics Engineering at Middlesex University, building upon my BEng in Mechatronics with a minor specialization in AI and Machine Learning. My background includes developing award-winning, patent-pending self-balancing systems and vision-based specimen handling platforms using high-precision depth sensing. I am particularly interested in narrowing the gap between "what robots can do" and "what we need them to do" through rigorous validation and system optimization.

When I'm not in the lab, you can find me on the field playing competitive cricket, football, or volleyball. I also enjoy the strategic challenge of a chess match and reading new books.

What drives me? The moment hardware, software, and intelligence converge perfectly.

Gesture Multi-Drone Control

AI · Custom NN · Webots · Robotics

Overview

A fully gesture-driven system to control 3 simulated DJI Mavic2Pro drones in Webots using only hand gestures captured by a laptop webcam. Two separate neural networks — both built entirely from scratch in NumPy — classify left-hand pose (which drone to select) and right-hand pose (what action to perform), with results written to a shared JSON file that each drone's controller reads every 8ms.

Key Achievements

  • Two-hand control scheme: Left hand selects drone(s) — 1/2/3 fingers or open palm for all. Right hand commands action — Takeoff, Land, Altitude Up/Down, Flip Forward, Hover
  • Custom 5-layer FC-NN from scratch: 35→64(ReLU)→32(ReLU)→16(Tanh)→N(Softmax), with L2 regularisation and cross-entropy loss — no PyTorch, no TensorFlow
  • Three optimisers implemented: Adam (first-order), L-BFGS (quasi-Newton, second-order), Nelder-Mead (zero-order gradient-free) — compared experimentally with 5-fold CV
  • 35-feature hand vector: 10 fingertip relative positions, 5 extension angles, 3 wrist orientation values, 2 palm centre offsets, extracted per hand from MediaPipe HandLandmarker (21 landmarks)
  • PID altitude control + motor mixing: P-controller (K_P=3.0) drives 4 motors per drone; mass compensation adjusts thrust proportionally when payload changes mid-flight
  • Flip state machine: 3-stage (Windup 160ms → Rotate 320ms → Recover 240ms) for safe mid-air forward flips at >1.5m altitude
  • Fail-safe: No gesture detected for >2s → drones hold position autonomously
  • Personally recorded 800–1,200 hand-gesture training samples across 10 gesture classes (4 left-hand, 6 right-hand)

Autonomous Mapping Robot

Embedded · C++ · Robotics

Overview

A dual-Arduino autonomous robot that maps a room using HC-SR04 ultrasonic sensors and encoder odometry, then executes a planned return journey with 5 equally-spaced scheduled stops. Clean Master/Slave architecture separates sensing from motor control across two Arduino Uno boards.

Key Achievements

  • Master (sensor board) sends commands <MAP>, <RUN>, <STOP>, <SLOW> to the Slave (motor board) via serial
  • Encoder-based odometry: 850 counts = 1 wheel revolution — minimum detectable movement ~7μm
  • Bridge detection via upward-facing HC-SR04: auto-slows to 50% speed through narrow passages, then resumes
  • Rear obstacle safety: unexpected obstacle triggers emergency stop with red LED indicator
  • DRV8833 motor driver with tunable LEFT_SPEED_CORRECTION constant for drift compensation
  • Configurable number of stops via NUM_INTERVALS constant — no re-wiring needed

Niryo NED2 Test Tube Sorting System

Robotics · RFID · ROS2 · Embedded

Overview

A fully autonomous three-component lab robotics system that combines mobile navigation, 6-DOF manipulation, and RFID identification to sort test tubes into the correct bins without any human intervention. TurtleBot4 transports loaded tube holders from a pickup point to the docking station, triggering the Niryo NED2 arm via HTTP REST API to pick each tube, scan it with an ESP32+PN532 RFID reader, and place it in the correct bin slot — all orchestrated through event-driven threading.

Key Achievements

  • Three-layer architecture: Hardware (NED2 + TurtleBot4 + ESP32), Software (ROS2 + PyNiryo + Flask), Integration (HTTP REST API + threading events + thread-safe state)
  • Flask REST API with 6 endpoints: /start_niryo, /rfid, /niryo_done, /status, /emergency_stop, /reset — enabling fully decoupled multi-robot coordination
  • Event-driven RFID sync: threading.Event() blocks the Niryo thread at the scanner position until the ESP32 sends the tag UID via WiFi HTTP GET, then unblocks to route to the correct bin
  • 4-bin Cartesian sorting: Niryo uses hardcoded joint-space waypoints (Approach → Place Approach → Place) at SLOW_SPEED (20%) for safe tube insertion; 4 UID-to-bin mappings in rfid_bin_map
  • ESP32 + PN532 RFID scanner: I2C on GPIO21/22, MIFARE ISO14443A, <5cm range — reads tag UID and fires HTTP GET to Flask server; Arduino C firmware with WiFi, Wire, HTTPClient libraries
  • TurtleBot4 coordinator: ROS2 Nav2-driven loop — navigate to pickup → dock → trigger Niryo → poll /niryo_done → undock → clear costmaps → repeat
  • Collision recovery: Auto-clears collision detected flag and retries movement; safe fallback to home position on critical errors; emergency stop via learning mode enable
  • 4 test tubes per cycle across 2 bins (A & B) with 2 slots each; post-dock costmap clearing + 2s AMCL settle prevents Nav2 localisation drift between cycles
